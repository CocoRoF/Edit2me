# 00. Vision

## 한 줄 요약

브라우저에서 PDF를 열어 텍스트와 페이지를 직접 편집하고 다운로드할 수 있는 도구. 모든 PDF 처리는 외부 라이브러리 없이 자체 구현한 엔진으로 한다.

## 왜 만드는가

기존 온라인 PDF 편집기는 (1) 광고/유료 모델, (2) 업로드된 파일의 보존/사용에 대한 불투명함, (3) 한국어/CJK 글꼴 처리의 빈약함이 문제다. Edit2me는 hr_blog2.0 인프라 위에 사용자가 통제 가능한 도구로 동작하며, 핵심 엔진을 직접 소유함으로써 이 문제들을 정면으로 다룬다.

## 목표 (Goals)

1. **MVP 5개 기능을 실사용 수준으로 동작**시킨다 (텍스트 편집/추가, 페이지 재배치/삭제, 병합).
2. **외부 PDF 라이브러리 0개**. 헤더부터 xref까지 직접 파싱하고 직렬화한다. (단, 압축은 표준 `zlib`/`CompressionStream` 허용)
3. **hr_blog2.0과 분리된 코드베이스, 통합된 배포** — 같은 docker-compose, 같은 nginx, 공유 MinIO. URL은 `/edit2me`.
4. **최소 의존성**. UI는 Next.js와 Tailwind만으로 끝낸다. 새 의존성은 ADR을 거친다.
5. **읽을 수 있는 코드**. PDF 처리 코드는 ISO 32000-1 섹션 번호를 주석으로 달아 명세 ↔ 구현이 1:1로 대응되도록 한다.

## 비목표 (Non-goals)

이번 사이클에서 의도적으로 하지 **않는** 것들. 나중에 다룰 수도 있지만 MVP 범위 밖이다.

- **암호화된 PDF의 편집** — 읽기는 가능할 수 있지만 편집/저장은 v1 범위 밖. 비밀번호가 걸린 PDF는 명시적 거부.
- **양식(form, AcroForm/XFA) 편집** — 인식해서 보존만 하고, 편집 UI는 안 만든다.
- **임의 글꼴로 새 텍스트 추가** — 새로 추가하는 텍스트는 *내장된 코어 14 글꼴* 또는 *사용자 업로드 TTF 1종*으로 제한. (PDF 텍스트 추가는 폰트 임베딩 문제와 직결되므로 범위를 좁힌다.)
- **OCR (스캔 이미지에서 텍스트 추출)** — 스캔 PDF는 텍스트 편집 불가, 페이지 단위 조작만 가능.
- **PDF/A·PDF/X 같은 사양 준수** — 출력은 일반 PDF 1.7로만 보장.
- **수식/표/벡터 그래픽 직접 편집** — 페이지 위 텍스트 객체에만 한정.
- **모바일 풀 지원** — 데스크톱/태블릿 가로 모드 우선. 모바일은 보기 + 페이지 조작만.
- **자체 사용자 시스템** — hr_blog2.0의 인증을 그대로 따른다 (일단은 익명 세션만).

## 성공 기준 (Definition of Done for MVP)

기능이 "되긴 하는" 것을 넘어, 다음을 충족해야 MVP로 본다.

| 기준 | 측정 방법 |
|---|---|
| **편집 호환성** | 테스트 코퍼스([`11-testing.md`](./11-testing.md)) 50개 PDF 중 텍스트 편집 후 모든 PDF 뷰어(Chrome, Acrobat, macOS Preview, Firefox)에서 정상 표시 ≥ 90%. |
| **페이지 조작 안정성** | 코퍼스 50개 모두 페이지 삭제/재배치/병합 후 페이지 트리 일관성 검증 통과 (자체 validator + qpdf `--check`로 교차검증, 단 qpdf는 *검증 도구로만* 사용 — 런타임 의존성 아님). |
| **렌더링 충실도** | 텍스트 위치 오차 ≤ ±2pt, 코어 글꼴 14종 100% 지원. 비ASCII는 v1에서 "보이기는 함" 수준 (정확한 metric은 v2). |
| **성능** | 50페이지/5MB PDF 기준: 첫 페이지 렌더 < 1.5s (서버), 페이지 조작 응답 < 200ms (클라이언트), 저장 < 3s. |
| **메모리** | 100MB PDF를 OOM 없이 처리 (스트리밍 파싱, 객체 lazy load). |
| **회귀 테스트** | 코어 파서/라이터 테스트 커버리지 ≥ 80% line, ≥ 70% branch. |

## 사용자 시나리오 (Top 3)

1. **"계약서에서 오타 하나 고치기"** — 사용자가 PDF를 업로드 → 1페이지에서 텍스트 클릭 → 인라인 편집 → 다운로드. 5초 내 완료가 목표.
2. **"여러 PDF에서 필요한 페이지만 모으기"** — 여러 파일 업로드 → 썸네일 보드에서 원하는 페이지만 드래그 → 순서 조정 → 새 PDF 다운로드.
3. **"발표 자료에서 일부 페이지 빼기"** — 업로드 → 썸네일에서 페이지 다중 선택 → 삭제 → 다운로드.

각 시나리오의 전체 UX 흐름은 [`07-ui-ux.md`](./07-ui-ux.md)에 있다.

## 핵심 설계 원칙

1. **Round-trip 보존이 최우선**. "편집하지 않은 부분"은 바이트 수준에서 가능한 한 그대로 유지한다. → Incremental update를 기본 저장 모드로 한다 ([`04-writer.md`](./04-writer.md) 참고).
2. **Lazy + Streaming**. 큰 PDF에서도 메모리에 전부 올리지 않는다. xref 우선, 객체 on-demand.
3. **표준 우선, 휴리스틱 명시**. 비표준 PDF에 대한 우회 처리는 반드시 주석에 사례를 남긴다.
4. **Server-side authority**. 최종 저장(직렬화)은 항상 서버에서 한다. 클라이언트는 *제안된 편집 연산*만 보내고, 서버가 검증·적용·재직렬화한다.
5. **연산은 데이터다**. 편집은 `{ op: "delete-page", index: 3 }` 같은 직렬화 가능한 연산 객체로 표현 → 실행 취소/재실행/감사 로그 무료.

## 구현 가능성에 대한 솔직한 평가

PDF 파서를 처음부터 짜는 것은 *가능하지만 큰 작업*이다. 본 프로젝트에서 가장 큰 리스크는 **렌더링**이다:

- 파싱·페이지 조작·병합·텍스트 추가는 명세를 따르면 결정론적으로 풀린다.
- **텍스트 편집**은 폰트(특히 임베디드 서브셋 + CMap)를 다뤄야 해서 난이도가 한 단계 위다.
- **렌더링**(페이지를 사용자에게 보여주기)은 그래픽 상태 머신, 폰트 글리프 디코딩, 색공간, 패턴까지 풀어야 해서 가장 어렵다.

→ 이 리스크는 [`12-risks.md`](./12-risks.md)에서 다루고, [`10-roadmap.md`](./10-roadmap.md)에서는 렌더링을 **가장 늦은 단계로 미루고 초기에는 텍스트-only 오버레이 + 페이지 썸네일**로 시작한다.
